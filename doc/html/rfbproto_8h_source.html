214 /*-----------------------------------------------------------------------------
215  * Protocol Version
216  *
217  * The server always sends 12 bytes to start which identifies the latest RFB
218  * protocol version number which it supports. These bytes are interpreted
219  * as a string of 12 ASCII characters in the format "RFB xxx.yyy\n" where
220  * xxx and yyy are the major and minor version numbers (for version 3.3
221  * this is "RFB 003.003\n").
222  *
223  * The client then replies with a similar 12-byte message giving the version
224  * number of the protocol which should actually be used (which may be different
225  * to that quoted by the server).
226  *
227  * It is intended that both clients and servers may provide some level of
228  * backwards compatibility by this mechanism. Servers in particular should
229  * attempt to provide backwards compatibility, and even forwards compatibility
230  * to some extent. For example if a client demands version 3.1 of the
231  * protocol, a 3.0 server can probably assume that by ignoring requests for
232  * encoding types it doesn't understand, everything will still work OK. This
233  * will probably not be the case for changes in the major version number.
234  *
235  * The format string below can be used in sprintf or sscanf to generate or
236  * decode the version string respectively.
237  */

663  * then the other bits are irrelevant; w*h pixel values follow (where w and h
664  * are the width and height of the tile). Otherwise the tile is encoded in a
665  * similar way to RRE, except that the position and size of each subrectangle
666  * can be specified in just two bytes. The other bits in the mask are as
667  * follows:
668  *
669  * BackgroundSpecified - if set, a pixel value follows which specifies
670  * the background colour for this tile. The first non-raw tile in a
671  * rectangle must have this bit set. If this bit isn't set then the
672  * background is the same as the last tile.
673  *
674  * ForegroundSpecified - if set, a pixel value follows which specifies
675  * the foreground colour to be used for all subrectangles in this tile.
676  * If this bit is set then the SubrectsColoured bit must be zero.
677  *
678  * AnySubrects - if set, a single byte follows giving the number of
679  * subrectangles following. If not set, there are no subrectangles (i.e.
680  * the whole tile is just solid background colour).
681  *
682  * SubrectsColoured - if set then each subrectangle is preceded by a pixel
683  * value giving the colour of that subrectangle. If not set, all
684  * subrectangles are the same colour, the foreground colour; if the
685  * ForegroundSpecified bit wasn't set then the foreground is the same as
686  * the last tile.
687  *
688  * The position and size of each subrectangle is specified in two bytes. The
689  * Pack macros below can be used to generate the two bytes from x, y, w, h,
690  * and the Extract macros can be used to extract the x, y, w, h values from

722 /*- - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - -
723  * Tight and TightPng Encoding.
724  *
725  *-- TightPng is like Tight but basic compression is not used, instead PNG
726  * data is sent.
727  *
728  *-- The first byte of each Tight-encoded rectangle is a "compression control
729  * byte". Its format is as follows (bit 0 is the least significant one):
730  *
731  * bit 0: if 1, then compression stream 0 should be reset;
732  * bit 1: if 1, then compression stream 1 should be reset;
733  * bit 2: if 1, then compression stream 2 should be reset;
734  * bit 3: if 1, then compression stream 3 should be reset;
735  * bits 7-4: if 1000 (0x08), then the compression type is "fill",
736  * if 1001 (0x09), then the compression type is "jpeg",
737  * (Tight only) if 1010 (0x0A), then the compression type is
738  * "basic" and no Zlib compression was used,
739  * (Tight only) if 1110 (0x0E), then the compression type is
740  * "basic", no Zlib compression was used, and a "filter id" byte
741  * follows this byte,
742  * (TightPng only) if 1010 (0x0A), then the compression type is
743  * "png",
744  * if 0xxx, then the compression type is "basic" and Zlib
745  * compression was used,
746  * values greater than 1010 are not valid.
747  *
748  * If the compression type is "basic" and Zlib compression was used, then bits
749  * 6..4 of the compression control byte (those xxx in 0xxx) specify the
750  * following:
751  *
752  * bits 5-4: decimal representation is the index of a particular zlib
753  * stream which should be used for decompressing the data;
754  * bit 6: if 1, then a "filter id" byte is following this byte.
755  *
756  *-- The data that follows after the compression control byte described
757  * above depends on the compression type ("fill", "jpeg", "png" or "basic").
758  *
759  *-- If the compression type is "fill", then the only pixel value follows, in
760  * client pixel format (see NOTE 1). This value applies to all pixels of the
761  * rectangle.
762  *
763  *-- If the compression type is "jpeg" or "png", the following data stream
764  * looks like this:
765  *
766  * 1..3 bytes: data size (N) in compact representation;
767  * N bytes: JPEG or PNG image.
768  *
769  * Data size is compactly represented in one, two or three bytes, according
770  * to the following scheme:
771  *
772  * 0xxxxxxx (for values 0..127)
773  * 1xxxxxxx 0yyyyyyy (for values 128..16383)
774  * 1xxxxxxx 1yyyyyyy zzzzzzzz (for values 16384..4194303)
775  *
776  * Here each character denotes one bit, xxxxxxx are the least significant 7
777  * bits of the value (bits 0-6), yyyyyyy are bits 7-13, and zzzzzzzz are the
778  * most significant 8 bits (bits 14-21). For example, decimal value 10000
779  * should be represented as two bytes: binary 10010000 01001110, or
780  * hexadecimal 90 4E.
781  *
782  *-- If the compression type is "basic" and bit 6 of the compression control
783  * byte was set to 1, then the next (second) byte specifies "filter id" which
784  * tells the decoder what filter type was used by the encoder to pre-process
785  * pixel data before the compression. The "filter id" byte can be one of the
786  * following:
787  *
788  * 0: no filter ("copy" filter);
789  * 1: "palette" filter;
790  * 2: "gradient" filter.
791  *
792  *-- If bit 6 of the compression control byte is set to 0 (no "filter id"
793  * byte), or if the filter id is 0, then raw pixel values in the client
794  * format (see NOTE 1) will be compressed. See below details on the
795  * compression.
796  *
797  *-- The "gradient" filter pre-processes pixel data with a simple algorithm
798  * which converts each color component to a difference between a "predicted"
799  * intensity and the actual intensity. Such a technique does not affect
800  * uncompressed data size, but helps to compress photo-like images better.
801  * Pseudo-code for converting intensities to differences is the following:
802  *
803  * P[i,j] := V[i-1,j] + V[i,j-1] - V[i-1,j-1];
804  * if (P[i,j] < 0) then P[i,j] := 0;
805  * if (P[i,j] > MAX) then P[i,j] := MAX;
806  * D[i,j] := V[i,j] - P[i,j];
807  *
808  * Here V[i,j] is the intensity of a color component for a pixel at
809  * coordinates (i,j). MAX is the maximum value of intensity for a color
810  * component.
811  *
812  *-- The "palette" filter converts true-color pixel data to indexed colors
813  * and a palette which can consist of 2..256 colors. If the number of colors
814  * is 2, then each pixel is encoded in 1 bit, otherwise 8 bits is used to
815  * encode one pixel. 1-bit encoding is performed such way that the most
816  * significant bits correspond to the leftmost pixels, and each raw of pixels
817  * is aligned to the byte boundary. When "palette" filter is used, the
818  * palette is sent before the pixel data. The palette begins with an unsigned
819  * byte which value is the number of colors in the palette minus 1 (i.e. 1
820  * means 2 colors, 255 means 256 colors in the palette). Then follows the
821  * palette itself which consist of pixel values in client pixel format (see
822  * NOTE 1).
823  *
824  *-- The pixel data is compressed using the zlib library. But if the data
825  * size after applying the filter but before the compression is less then 12,
826  * then the data is sent as is, uncompressed. Four separate zlib streams
827  * (0..3) can be used and the decoder should read the actual stream id from
828  * the compression control byte (see NOTE 2).
829  *
830  * If the compression is not used, then the pixel data is sent as is,
831  * otherwise the data stream looks like this:
832  *
833  * 1..3 bytes: data size (N) in compact representation;
834  * N bytes: zlib-compressed data.
835  *
836  * Data size is compactly represented in one, two or three bytes, just like
837  * in the "jpeg" compression method (see above).
838  *
839  *-- NOTE 1. If the color depth is 24, and all three color components are
840  * 8-bit wide, then one pixel in Tight encoding is always represented by
841  * three bytes, where the first byte is red component, the second byte is
842  * green component, and the third byte is blue component of the pixel color
843  * value. This applies to colors in palettes as well.
844  *
845  *-- NOTE 2. The decoder must reset compression streams' states before
846  * decoding the rectangle, if some of bits 0,1,2,3 in the compression control
847  * byte are set to 1. Note that the decoder must reset zlib streams even if
848  * the compression type is "fill", "jpeg" or "png".
849  *
850  *-- NOTE 3. The "gradient" filter and "jpeg" compression may be used only
851  * when bits-per-pixel value is either 16 or 32, not 8.
852  *
853  *-- NOTE 4. The width of any Tight-encoded rectangle cannot exceed 2048
854  * pixels. If a rectangle is wider, it must be split into several rectangles
855  * and each one should be encoded separately.
856  *
857  */
858
